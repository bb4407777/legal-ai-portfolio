# AI 与法律数据实践

> 从真实法律场景出发，定义数据需求，驱动 AI 产品迭代。

---

## 使用规模

- **月均 DeepSeek Pro token 消耗：** 约 38 亿 tokens
- **月均 DeepSeek Flash token 消耗：** 约 38 亿 tokens
- **合计月均：** 约 76 亿 tokens
- **主要使用模型：** DeepSeek + Claude（通过 Claude Code 自定义后端接入）

这不是为了消耗而消耗。每一笔 token 都对应一个真实的办案需求。

## 应用场景

### 法律检索辅助

大模型在法律检索中显著提升了效率。通过对接后的 RAG 管道，实现对法律条文、司法解释、指导案例的语义检索，而非传统的关键词匹配。

### 文书生成与审查

- 起诉状、答辩状、代理词的初稿生成
- 合同审查要点提取
- 法律意见书的事实梳理与论证生成

关键洞察：**法律文书生成的核心瓶颈不是模型能力，而是数据质量**——准确的事实数据、清晰的时间线、完整的当事人信息，这些才是高质量法律文书的基石。

### 案件分析与推理

- 争议焦点自动归纳
- 类案推荐与裁判趋势分析
- 证据链完整性检查

### Agent 工作流实践

探索法律 Agent 的多步骤工作流设计：

```
用户输入（自然语言）
       │
       ▼
意图识别（咨询/立案/查案/文书）
       │
       ▼
工具调用（检索法条/检索案件/查询当事人）
       │
       ▼
信息整合与推理
       │
       ▼
结果输出
```

## 关键实践

### Claude Code 对接 DeepSeek

独立完成 Claude Code 的后端配置，将默认模型从 Claude 切换为 DeepSeek，实现：
- 使用 DeepSeek 的推理能力替代 Claude 处理中文法律场景
- 通过自定义 API 端点集成到现有工具链
- 持续对比两家模型在法律场景的表现差异

### Prompt Engineering 实践

基于真实案件场景，设计了多套 Prompt 模板：
- **文书生成模板：** 包含案件事实要素、格式规范、引用标准的结构化 Prompt
- **法律分析模板：** 多步推理链，引导模型逐步分析争议焦点
- **数据提取模板：** 从非结构化文本中提取结构化案件信息

### 评估与反思

长期使用 AI 在法律场景中获得的关键结论：

| 发现 | 说明 |
|------|------|
| 幻觉是最大的风险 | 法律场景中一条虚构的法条可以导致一个案子输掉 |
| 中文法律推理优于英文翻译 | DeepSeek 在法律推理上优于从英文翻译的法律分析 |
| 长上下文是关键 | 很多案子需要处理数百页材料，窗口大小直接影响可用性 |
| 结构化的输入决定输出质量 | 同样的模型，给结构化数据和给非结构化文本，输出质量天差地别 |

## 为什么这些经验对法律 AI 数据产品有价值

法律 AI 的核心瓶颈从来不是模型能力——模型的能力每半年翻一番。核心瓶颈是：

1. **法律数据的质量与结构化程度**
2. **对法律业务场景的深刻理解**
3. **评估体系——如何判断模型「懂法」**

我做了一线律师、搭了业务系统、每天都在用 AI 办案。

这恰好是理解上面三个瓶颈的最佳角度。
